# OCR 图像文本加载器实验报告

## 一、架构设计

### 1.1 ImageOCRReader 在 LlamaIndex 中的位置

```
图像文件 (.png/.jpg/.pdf扫描件)
         │
         ▼
  ┌──────────────┐
  │ ImageOCRReader│   ← 自定义 BaseReader（本实验实现）
  │  (PP-OCR v5) │
  └──────┬───────┘
         │ Document(text, metadata)
         ▼
  ┌──────────────┐
  │ VectorStore  │   ← LlamaIndex 标准组件
  │   Index      │
  └──────┬───────┘
         │
         ▼
  ┌──────────────┐
  │ Query Engine │   ← 检索 + LLM 生成回答
  └──────────────┘
```

**设计思路**：ImageOCRReader 作为数据加载层，将非结构化的图像数据转化为 LlamaIndex 统一的 Document 格式，使其能够无缝接入后续的索引和检索流程。这是 LlamaIndex "Reader → Document → Index → Query" 标准流程的自然扩展。

## 二、核心代码说明

### 2.1 ImageOCRReader 类设计

```python
class ImageOCRReader(BaseReader):
    def __init__(self, lang='ch', use_gpu=False, **kwargs)
    def load_data(self, file: Union[str, List[str]]) -> List[Document]
    def load_data_from_dir(self, dir_path: str) -> List[Document]
```

| 方法 | 职责 |
|:---|:---|
| `__init__` | 配置语言、GPU 等参数，采用延迟初始化避免导入时加载模型 |
| `load_data` | 核心方法：调用 PP-OCR 提取文本，封装为 Document |
| `load_data_from_dir` | 扩展方法：批量扫描目录下的图像文件 |

### 2.2 关键设计决策

- **延迟初始化 OCR 模型**：PaddleOCR 首次实例化时需加载模型权重（约 100MB+），放在 `__init__` 中会导致 `import` 时就加载。通过 `_get_ocr()` 延迟到首次 `load_data` 时才初始化。
- **文本块格式化**：每个识别结果标注序号和置信度，格式为 `[Text Block N] (conf: X.XX): 文本内容`，便于追踪和调试。
- **置信度统计**：计算所有文本块的平均置信度并写入 metadata，为后续质量评估提供参考。

### 2.3 Document 元数据设计

| 字段 | 类型 | 用途 |
|:---|:---|:---|
| `image_path` | str | 来源追溯，定位原始图像 |
| `ocr_model` | str | 版本记录，便于对比不同 OCR 引擎 |
| `language` | str | 语言标识，支持多语言场景 |
| `num_text_blocks` | int | 文本块数量，粗略反映图像信息密度 |
| `avg_confidence` | float | 平均置信度，用于质量评估和过滤 |

## 三、OCR 效果评估

### 3.1 测试图像

| 类型 | 文件 | 特点 |
|:---|:---|:---|
| 通用 OCR | `general_ocr_002.png` | 混合中英文、多字体、含数字和符号 |

### 3.2 识别效果分析

**扫描文档 / 清晰文本图像**：
- PP-OCR 对印刷体中文的识别准确率极高（> 98%）
- 数字和英文字母识别同样准确
- 平均置信度通常在 0.85 以上

**屏幕截图（含 UI 文字）**：
- 界面中的菜单文字、按钮文本识别良好
- 小字号文字（< 12px）可能遗漏或识别错误
- 图标旁的文字可能有轻微偏差

**自然场景图（路牌、广告牌）**：
- 倾斜、透视变形的文字识别能力有限
- 复杂背景（如文字与图片重叠）会降低准确率
- 艺术字体、手写体识别效果较差

## 四、错误案例与分析

### 4.1 常见错误类型

| 错误类型 | 原因 | 示例 |
|:---|:---|:---|
| 字符混淆 | 形状相近的字难以区分 | "0" 被识别为 "O"，"1" 被识别为 "l" |
| 漏字 | 字体过小或模糊 | 小字号注释被跳过 |
| 多余空格 | 布局不规则 | 表格列间距导致额外空格 |
| 乱码 | 特殊符号或公式 | 数学公式无法正确识别 |

### 4.2 改进建议

1. **图像预处理**：对模糊图像先做锐化/去噪处理，对倾斜图像做矫正
2. **模型选择**：对高精度场景使用 PP-OCRv5_server 模型（更大但更准确）
3. **后处理**：结合语言模型对 OCR 结果进行纠错

## 五、Document 封装合理性讨论

### 5.1 文本拼接方式

当前实现将所有文本块按 OCR 检测顺序用换行符拼接，这种方式：

- ✅ **优点**：简单直接，保留文本的阅读顺序
- ❌ **局限**：丢失了空间结构信息（如表格的行列关系）

**改进方案**：对于表格等结构化内容，可以检测文本块的坐标关系，判断是否属于同一表格，再用 Markdown 表格格式拼接，这样能更好地保留结构信息。

### 5.2 元数据设计

当前元数据设计合理：
- `avg_confidence` 可以用于过滤低质量 OCR 结果（如 < 0.6 的直接丢弃）
- `num_text_blocks` 帮助判断图像的信息密度
- `image_path` 支持结果溯源

**可扩展字段**：可加入 `ocr_time`（耗时）、`resolution`（图像分辨率），便于性能分析。

## 六、局限性与改进建议

### 6.1 当前局限

1. **无空间结构保留**：表格、多栏排版的结构信息丢失
2. **无 PDF 支持**：只能处理单张图像，无法直接处理多页 PDF
3. **无批量可视化**：缺少 OCR 检测框的可视化输出
4. **依赖模型下载**：首次运行需下载 PaddleOCR 模型

### 6.2 改进建议

| 改进方向 | 方案 | 收益 |
|:---|:---|:---|
| 表格识别 | 使用 PP-Structure 替代基础 OCR | 保留表格结构，输出 Markdown 格式 |
| PDF 支持 | 每页转图像后 OCR，拼接为完整文档 | 支持扫描件 PDF |
| 可视化 | 用 OpenCV 在原图上绘制检测框 | 直观查看识别效果 |
| 质量过滤 | 根据 `avg_confidence` 阈值过滤低质量结果 | 提高索引质量 |
| 并行处理 | 多线程/多进程并行 OCR 多张图像 | 提升批量处理速度 |
